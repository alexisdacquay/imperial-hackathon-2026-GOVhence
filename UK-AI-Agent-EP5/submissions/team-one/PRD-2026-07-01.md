## Process (authoritative)

*Verbatim, as specified by the project owner on 2026-07-01. Do not reword.*

- user sends a message
- Main agent GOVhence receives message and user details (name, profile with role and department)
- GOVhence ask the classifier LLM to classify the message with tags (e.g. "where is the best sandwich"
  --> gets labelled with whatever the LLM classify, for example "food" tag, but also assigns the user
  details and RBAC/ACL from the profile to the message, e.g. "driver" (role) and "logistics" (department)
- the Judge LLM is triggered by GOVhence with message + tags
- Judge LLM decides whether 3 action: Read memory (script, strictly no LLM), Write memory (LLM), forward
  to the Responder LLM
- Judge LLM can do all three, but the first gate is "is this message worth checking for memory?". For
  example "Where is the best sandwich?" might be worth accessing memory to check whether we have in memory
  something related to where, or best sandwich shops around. If YES (it's worthy to check memory), then
  pass the message + tags to the Memory retriever: Bouncer.
- Bouncer is a script very strict about tags. no LLM. If match -> retrieve memory. Strict security
  matching tags and user access inheritence
- if the Judge LLM decide it's not memory-access worthy (e.g. just a "Hi"), then pass directly to the
  responder
- Judge MUST wait for the bouncer to get back with the memory retrieval response, either with nothing
  (there was no memory match), or the memory content (match success). The Judge must wait for the Bouncer
  script response. then forward to the responder LLM.
- in parallel, regardless of the memory read/access decision, there is AFTERWARDS (to not slow down the
  read and forwarding to the responder, the Judge as a next step for that SAME message (and tags) will
  judge whether it's worthy of writing memory about it. For example, if the user has a message "no, not
  in Tokyo, I want a sandwich where I am now in London", then the Judge LLM would probably judge that the
  user provides useful information or feedback about where he is. The Judge LLM passes the candidate
  content to memorise (what it decided was useful) and the tags.
- The writing into memory is done by the Memoriser LLM. it uses the "memory candidate" and tags received
  from the Judge LLM. (roadmap : can have a second run of verifications, add to a ROADMAP.md). That LLM
  should search for existing tags, to avoid duplication.

## Naming

- **Bouncer** — the memory retriever: a strict, **script-only (no LLM)** tag matcher that also enforces
  RBAC/ACL and user access inheritance. This is the name of record (the code already uses `bouncer`).
- *"DeterminExtractor"* — **deprecated alias** for the Bouncer used in the older `PRD.md`. Same
  component; use **Bouncer** going forward.

## What carried over from PRD.md (still useful — cross-checked, not contradicted)

These are constraints (not process), so the new Judge-centric flow does not change them. Restated here
so this file is self-sufficient; verify against `PRD.md` if you need the full rationale.

- **Open-weight models only** (Llama/Qwen/Mistral via BasedAPIs) — no closed models anywhere in the loop.
- **No LLM in the access decision.** The **Bouncer** is the sole access gate and is script-only,
  deterministic, fail-closed. LLMs classify / judge / memorise / respond — they never *gate* access.
- **Audit** — every retrieval decision (ALLOW + DENY) is logged; append-only, tamper-evident, complete.
- **Async memory-write** — the write path (Judge → Memoriser → store) runs in parallel and must **not**
  block the user's answer. Reading memory is the first-stage step *before* the Responder.
- **Tag hygiene** — reuse existing tags; single-word or hyphenated; no near-duplicate proliferation
  (`location` vs `Location`, `weekend` vs `week-end`).
- **Latency** — target sub-200ms P99 for permission checks.
- **Portability** — runs identically on Windows + macOS.
- **Fail-closed / RBAC-ACL** — untagged or ambiguous content denies; every read scoped to the user's
  permitted set.

## Superseded by this document (what changed vs PRD.md)

A reader of the old `PRD.md` should note these deliberate changes:

1. **Judge is the decision hub.** The Judge LLM decides among **3 actions** — Read memory (script/no-LLM),
   Write memory (LLM), forward to Responder — and *can do all three*. (Old PRD framed GOVhence as the
   orchestrator with the Judge only assessing usefulness.)
2. **First gate = "is this worth checking memory at all?"** A "Hi" skips retrieval and goes straight to
   the Responder. (Old PRD step 6 said *every* message is a retrieval candidate — that is superseded.)
3. **Classifier binds RBAC/ACL to the message** (role + department + access), explicitly.
4. **Judge triggers the Bouncer and MUST wait** for its response (nothing, or matched content) before
   forwarding to the Responder — a synchronous handoff.
5. **Judge owns write-candidate selection.** After the read/forward path, the same Judge re-judges the
   same message+tags for write-worthiness and hands the chosen *memory candidate + tags* to the Memoriser.
6. **Memoriser is leaner** — it writes the candidate + reuses existing tags to avoid duplication. A
   second-pass verification is **roadmap** (see ROADMAP.md), not part of this spec.
